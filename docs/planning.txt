================================================================================
1. AIM
================================================================================
Build an end-to-end Stage 1 vertical slice of VoiceShield that:

  - Captures live (or file-based) call audio at 16 kHz mono.
  - Processes it in 500 ms chunks held in a 10-second rolling buffer.
  - Applies an SNR quality gate (per architecture sec. 6).
  - Extracts forensic features:
        linear / mel / CQT spectrograms,
        phase discontinuity heatmap,
        pitch contour + smoothness,
        spectral flux,
        MFCC + delta + delta-delta,
        subband energy timeline (sec. 7.1 - 7.8).
  - Runs an AASIST-style anti-spoofing classifier on rolling windows and
    emits a synthetic-risk score in [0.0, 1.0] (sec. 8).
  - Maps the score to a colour-coded state: Green / Amber / Red, plus Grey
    from the SNR gate (sec. 8 + sec. 14).
  - Maintains a rolling artifact timeline (one entry per 500 ms chunk,
    sec. 12).
  - Streams current state and timeline over WebSocket / REST to a minimal
    advisory dashboard (sec. 15).
  - Flags a synthetic / cloned / TTS voice as Amber by ~5 s and Red by 8-10 s
    (sec. 5 - "10-second alert requirement").

Everything beyond this - Stage 2 ensemble, replay detection, full evidence
export with SHA-256, SIP/VoIP integration, Indian phoneme analysis - is
EXPLICITLY OUT OF SCOPE for this plan.


================================================================================
2. GOAL / ACCEPTANCE CRITERIA  (must all pass to call Stage 1 "done")
================================================================================
G1.  End-to-end pipeline runs from one command on a CPU-only laptop.
       No GPU required, no network calls for inference.

G2.  Genuine human speech sample -> steady Green state, mean score < 0.30.

G3.  A known synthetic / TTS / cloned WAV sample -> state reaches Amber by
       <= 5 s and Red by <= 10 s of audio (trigger_time <= 10.0 s).

G4.  Per-chunk latency (capture -> score -> dashboard push) is < 200 ms on
       a typical CPU (p95).

G5.  SNR gate correctly degrades to "Grey" on a silent or very noisy input
       (SNR < 8 dB) and is independent of the classifier score.

G6.  Artifact timeline contains exactly one entry per 500 ms chunk with
       fields { time, score, state, snr_db, top_artifact }.

G7.  Browser-side dashboard updates at least every 500 ms over WebSocket and
       reflects state changes within 1 chunk.

G8.  Every module ships with unit tests; suite green via `pytest -q`.

G9.  All acceptance tests TEST-E2E10.1 ... TEST-E2E10.5 pass on a clean
       machine following the README.

G10. No call recording persisted to disk beyond the 10-second in-memory
       rolling buffer, unless a manual "save for audit" flag is set (the
       privacy guarantee from sec. 5).


================================================================================
3. SCOPE
================================================================================
IN  (Stage 1 / MVP):
    - Live mic capture + WAV file capture.
    - 500 ms chunking, 10 s rolling buffer.
    - SNR quality gate (Normal / Reduced / Grey).
    - All Stage 1 forensic features (sec. 7).
    - Stage 1 classifier (AASIST-style) + a rule-based fallback.
    - Score -> state mapper (Green / Amber / Red / Grey).
    - Artifact timeline data structure + JSON export of timeline only.
    - FastAPI backend with WebSocket + REST endpoints.
    - Minimal single-page HTML/JS advisory dashboard.
    - pytest suite (unit + integration + end-to-end).

OUT (deferred to Stage 2 / Enhanced Prototype):
    - Stage 2 ensemble (WavLM, Spectrogram CNN, Phase/Pitch classifier).
    - Replay detection (reverb, freq response, double compression, bg mismatch).
    - Full evidence package export (spectrogram PNG, phase heatmap PNG,
      SHA-256 audio hash, model-scores JSON envelope) - sec. 13.
    - Risk fusion formula across Stage 1+2+Replay (sec. 14).
    - SIP / VoIP / PBX / telephony middleware integration.
    - Multi-tenant call orchestration, K8s deployment, autoscaling.
    - Indian phoneme analysis, federated learning, fraud graph.

OUT (research / future roadmap):
    - Physiological liveness veto gate.
    - National voice trust layer / cross-bank fraud graph.
    - Secure MPC model sharing.

4. TECH STACK (locked for Stage 1)
================================================================================
Language        : Python 3.11+
Audio I/O       : sounddevice (live), soundfile (WAV), librosa
Signal / DSP    : numpy, scipy, librosa  (STFT, mel, CQT, MFCC, pYIN F0)
ML              : torch (CPU build), torchaudio, onnxruntime (optional)
Stage 1 model   : AASIST-L  (clovaai/aasist) - pre-trained weights
                   Fallback: lightweight rule-based scorer over phase+pitch+flux
Backend         : FastAPI + Uvicorn, websockets
Frontend        : Plain HTML + vanilla JS + Canvas/SVG (no React in Stage 1)
Testing         : pytest, pytest-asyncio, hypothesis (property tests)
Tooling         : Poetry or pip-tools, ruff, black, pre-commit

        [INPUTS]   what the task consumes.
        [OUTPUTS]  what the task produces.
        [DOD]      definition-of-done; concrete observable.

        [DEPS]     upstream tasks that must already be done.

Tasks are atomic. If a task takes > 1 working day, split it.

9. RISK REGISTER
================================================================================
R1.  AASIST-L weights licensing - confirm permitted use for the hackathon
     demo. Fallback: T5.5 rule-based classifier always usable.

R2.  Live mic capture is platform-dependent (PortAudio).
     Mitigation: keep FileSource as the primary CI path, mic as demo-only.

R3.  Synthetic / cloned voice may not produce strong artifacts on top-tier
     TTS systems.  Acceptance tests must use a curated, known-detectable
     sample for the hackathon demo.  Future Stage 2 ensemble is what
     hardens this.

R4.  Latency budget (G4) may be tight on low-end CPUs.
     Mitigation: ONNX-export AASIST and benchmark in T5.3.

R5.  pYIN F0 can be slow.  If T4.5/T4.6 dominate latency, switch to
     librosa.pyin with shorter frames or use a lightweight autocorrelation
     pitch tracker.

R6.  Hysteresis (T7.2) can suppress true positives.
     Mitigation: pin the rule to 2 consecutive chunks for AMBER escalation
     only; allow immediate RED if score > 0.85 in a single chunk.

10. STAGE-2 HAND-OFF NOTES (out-of-scope here but designed-for)
================================================================================
The following extension points must be preserved by Stage 1 implementers so
Stage 2 can plug in without refactoring:

  - classifier package exposes a Scorer protocol with .score(audio) -> float.
    Stage 2 adds WavLMScorer, SpectrogramCNNScorer, PhasePitchScorer
    implementing the same protocol.

  - pipeline.state_engine.compute_final accepts an arbitrary dict of
    component scores, with stage1 the only key in Stage 1.  Risk fusion in
    Stage 2 simply passes more keys.

  - TimelineEntry already carries snr_db and top_artifact, sufficient for
    Stage 2 evidence enrichment.

  - The /risk/* REST contract is versioned (/v1/); Stage 2 adds /v2/
    without breaking Stage 1 clients.

  - All audio handled is the same 16 kHz mono float32 contract end-to-end.
